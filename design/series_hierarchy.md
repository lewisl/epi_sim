# Series Hierarchy And Selection

This note documents the current `AllSeries` / `SeriesGroup` storage model in
`src/series.h` and `src/series.cpp`. It is separate from `PopData`
serialization. `PopData` column output converts individual person fields to
strings; series output works with integer count vectors that are already stored
by simulation day.

## Top-Level Series Container

`AllSeries` owns six named `SeriesGroup` instances:

```cpp
struct AllSeries {
    SeriesGroup now_status, new_status;
    SeriesGroup now_vax,    new_vax;
    SeriesGroup now_variant, new_variant;
    size_t day_cnt;
};
```

Each field answers two questions:

1. Which event family is this? `status`, `vax`, or `variant`.
2. Is this a stock count or a same-day event count? `now_*` or `new_*`.

`now_*` series are stock/current counts. They are carried forward from the
previous day at the start of each day and then adjusted by state changes.

`new_*` series are event/flow counts. They are not carried forward. They only
record events that happened on that day.

## SeriesGroup Physical Storage

`SeriesGroup` has one real nested storage member:

```cpp
struct SeriesGroup {
    using BucketArray = std::array<std::vector<int>, size_t(AgeBucket::COUNT)>;
    using RingArray   = std::vector<BucketArray>;

    std::vector<RingArray> subjects;
    size_t day_cnt{};
    size_t n_rings{};
};
```

The `using` declarations are type aliases only. They do not declare members.
The actual hierarchy lives inside `subjects`.

Expanded conceptually:

```text
subjects
└── [subject_idx]      vector index: status, vaccine, or variant value
    └── [ring_id]      vector index: 0 = RING_ALL, 1..N = real rings
        └── [bucket]   array index: AgeBucket enum value
            └── [day]  vector index: simulation day
                = int count
```

In C++ indexing form:

```cpp
subjects[subject_idx][ring_id][size_t(bucket)][day]
```

The helper accessor hides that full expression:

```cpp
series.now_status.at(subject_idx, bucket, ring_id)[day]
```

The innermost `std::vector<int>` is the day series. It is not a vector of
statuses, variants, or vaccines. Status, variant, and vaccine are represented by
the outer `subject_idx`.

Day vectors have size `day_cnt + 1`. Day 0 is unused. Valid days are
`1..day_cnt`.

## Meaning Of subject_idx

The meaning of `subject_idx` depends on which `SeriesGroup` is being accessed.

```text
now_status.subjects[status_idx][ring][bucket][day]
new_status.subjects[status_idx][ring][bucket][day]

now_vax.subjects[vax_idx][ring][bucket][day]
new_vax.subjects[vax_idx][ring][bucket][day]

now_variant.subjects[variant_idx][ring][bucket][day]
new_variant.subjects[variant_idx][ring][bucket][day]
```

For status groups, `subject_idx` is a `Status` value such as
`uint8_t(INFECTIOUS)` or `uint8_t(DEAD)`.

For vaccine groups, `subject_idx` is a `Vax` value such as `uint8_t(Vax{1})`.

For variant groups, `subject_idx` is a `Variant` value such as
`uint8_t(Variant{2})`.

The corresponding names are held in static `names` members:

```text
Status::names
Vax::names
Variant::names
```

For example, `Status::names` is the static `names` member of the `Status`
struct. It contains base trait names such as `unexposed`, `infectious`,
`recovered`, and `dead`. It does not contain `now_` or `new_`; those are series
selection prefixes added by the series layer.

## Ring And Age Bucket Dimensions

Ring index `0` is reserved:

```cpp
inline constexpr uint8_t RING_ALL = 0;
```

`RING_ALL` stores the aggregate across all real rings. Real rings use indices
`1..N`, matching their positions in `Ring::names`.

Age buckets are the `AgeBucket` enum:

```text
total
age0_19
age20_39
age40_59
age60_79
age80_up
```

The `total` bucket stores the aggregate across age groups for the selected
subject and ring.

## Where Values Are Generated

Values are generated by calls to:

```cpp
SeriesGroup::update(uint8_t subject_idx,
                    uint8_t ring,
                    Agegrp agegrp,
                    size_t day,
                    int change);
```

`update` converts the person's `Agegrp` to an `AgeBucket`, then applies the
same `change` to four places:

```text
specific ring, specific age bucket
specific ring, total age bucket
RING_ALL, specific age bucket
RING_ALL, total age bucket
```

The `RING_ALL` writes are skipped when the input ring is already `RING_ALL`, to
avoid double-counting.

This means a single event updates both detailed and aggregate views at the time
the event happens. Serialization and plotting do not recompute totals; they only
select existing day vectors.

The main disease state transitions call `update` from `AgentView` methods in
`src/disease_modeling.cpp`:

```cpp
person.make_sick(...)
person.make_well(...)
person.make_dead(...)
```

For example, `make_sick` increments:

```text
new_status[INFECTIOUS]
now_status[INFECTIOUS]
new_variant[var]
now_variant[var]
```

and decrements the previous stock status, usually:

```text
now_status[UNEXPOSED]
```

or:

```text
now_status[RECOVERED]
```

Vaccination updates similarly call `series.new_vax.update(...)` and
`series.now_vax.update(...)` from `src/vaccination.cpp`.

At the start of each simulation day, `AllSeries::init_history_series(day)`
carries forward only the stock groups:

```text
now_status
now_vax
now_variant
```

It does not carry forward `new_status`, `new_vax`, or `new_variant`.

## Text Selection Layer

Output and plotting do not index `subjects` directly. They use:

```cpp
struct SeriesSelection {
    std::string name;
    std::string bucket;
    std::string ring = "";
};
```

Each element is resolved independently before a physical day vector is selected.

### bucket

`bucket` is resolved by:

```cpp
age_bucket_from_string(sel.bucket)
```

The result is an `AgeBucket` enum value. Unknown bucket strings make the
selection invalid.

### ring

`ring` is resolved by:

```cpp
ring_id_from_token(sel.ring)
```

Rules:

```text
""          -> RING_ALL
"2"         -> literal ring id 2, if in range
"ring_2"    -> lookup in Ring::names
unknown     -> invalid selection
```

### name

`name` is resolved by:

```cpp
resolve_series(series, sel.name, bucket, ring)
```

For status series, `resolve_series` loops through `Status::names` and builds
candidate public names by adding `now_` and `new_`:

```text
"now_" + Status::names[i]
"new_" + Status::names[i]
```

So if `Status::names[INFECTIOUS] == "infectious"`, then:

```text
now_infectious -> series.now_status.at(INFECTIOUS, bucket, ring)
new_infectious -> series.new_status.at(INFECTIOUS, bucket, ring)
```

For variant series, names contain a group and a trait name:

```text
now_variant:delta
new_variant:delta
```

The text after the colon is looked up in `Variant::names`. The matching index is
used as the subject index into `now_variant` or `new_variant`.

For vaccine series, names work the same way:

```text
now_vax:pfizer
new_vax:pfizer
```

The text after the colon is looked up in `Vax::names`.

There are also aggregate vaccine selections:

```text
now_vaccinated
new_vaccinated
```

These do not correspond to one physical subject. `resolve_series` builds a
temporary day vector by summing all non-`none` vaccine subjects.

## Example

Suppose a person in `ring_1`, age group `age20_39`, becomes infectious with the
`delta` variant on day 7.

The disease transition calls:

```cpp
series.new_status.update(INFECTIOUS, ring_id, agegrp, 7, 1);
series.now_status.update(INFECTIOUS, ring_id, agegrp, 7, 1);
series.new_variant.update(delta_idx, ring_id, agegrp, 7, 1);
series.now_variant.update(delta_idx, ring_id, agegrp, 7, 1);
```

For `series.now_status.update(INFECTIOUS, ring_1, age20_39, 7, 1)`, the
physical writes are:

```text
now_status.subjects[INFECTIOUS][ring_1][age20_39][7] += 1
now_status.subjects[INFECTIOUS][ring_1][total][7]    += 1
now_status.subjects[INFECTIOUS][RING_ALL][age20_39][7] += 1
now_status.subjects[INFECTIOUS][RING_ALL][total][7]    += 1
```

A later output selection:

```cpp
{"now_infectious", "age20_39", "ring_1"}
```

resolves as:

```text
name   "now_infectious" -> now_status, subject INFECTIOUS
bucket "age20_39"       -> AgeBucket::age20_39
ring   "ring_1"         -> ring id for ring_1
```

and selects this physical day vector:

```cpp
series.now_status.at(uint8_t(INFECTIOUS),
                     AgeBucket::age20_39,
                     ring_id_for_ring_1)
```

The returned value is a `std::vector<int>` indexed by day. Output code reads
days `1..series.day_cnt`.

## Output Consumers

`print_selected_series`, `serialize_selected_series`, and `seriesplot` all use
the same selection model:

```text
SeriesSelection
-> bucket/ring/name resolution
-> vector<int> counts by day
```

They should continue to share the selection and resolution path, but their final
rendering is different:

```text
print_selected_series      human-readable terminal table
serialize_selected_series  CSV file
seriesplot                 Plotly JSON traces
```

The useful unification point is therefore a shared "resolve selected columns"
helper, not one combined output function.
