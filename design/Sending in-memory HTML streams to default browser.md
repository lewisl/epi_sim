# claude response

Short answer: no, not directly — browsers consume files or URLs, not stdin. But there's a clean solution that's better than temp files: a **one-shot localhost HTTP server**.

The idea: bind to `localhost:0` (OS picks the port), fire `open http://localhost:PORT`, serve the HTML over the socket once, then shut down. No files written, no cleanup needed, works with any browser.

```cpp
#include <sys/socket.h>
#include <netinet/in.h>
#include <unistd.h>
#include <string>
#include <string_view>
#include <thread>
#include <cstdlib>

// Serve html from memory to the default browser via a one-shot local HTTP server.
// Non-blocking: the server runs on a detached thread and dies after one request.
bool open_html_in_browser(std::string html) {  // take by value — thread captures it

    int sock = socket(AF_INET, SOCK_STREAM, 0);
    if (sock < 0) return false;

    int opt = 1;
    setsockopt(sock, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

    sockaddr_in addr{};
    addr.sin_family      = AF_INET;
    addr.sin_addr.s_addr = htonl(INADDR_LOOPBACK);
    addr.sin_port        = 0;  // let OS pick a free port

    if (bind(sock, reinterpret_cast<sockaddr*>(&addr), sizeof(addr)) < 0) {
        close(sock); return false;
    }

    socklen_t addrlen = sizeof(addr);
    getsockname(sock, reinterpret_cast<sockaddr*>(&addr), &addrlen);
    int port = ntohs(addr.sin_port);

    if (listen(sock, 1) < 0) {
        close(sock); return false;
    }

    // Open browser before blocking on accept — connection will queue.
    std::string url = "http://localhost:" + std::to_string(port);
#if defined(__APPLE__)
    std::system(("open \"" + url + "\"").c_str());
#elif defined(_WIN32)
    std::system(("start \"\" \"" + url + "\"").c_str());
#else
    std::system(("xdg-open \"" + url + "\"").c_str());
#endif

    std::thread([sock, html = std::move(html)]() mutable {
        // Browsers often make 2+ requests (favicon etc.). Serve until first
        // GET that isn't /favicon.ico, then close the listen socket.
        while (true) {
            int client = accept(sock, nullptr, nullptr);
            if (client < 0) break;

            // Read enough of the request to inspect the path.
            char buf[4096]{};
            recv(client, buf, sizeof(buf) - 1, 0);
            std::string_view req(buf);

            bool is_favicon = req.find("GET /favicon") != std::string_view::npos;

            if (is_favicon) {
                // 404 and keep listening.
                std::string_view not_found =
                    "HTTP/1.1 404 Not Found\r\nContent-Length: 0\r\nConnection: close\r\n\r\n";
                send(client, not_found.data(), not_found.size(), 0);
                close(client);
                continue;
            }

            // Serve the HTML.
            std::string header =
                "HTTP/1.1 200 OK\r\n"
                "Content-Type: text/html; charset=utf-8\r\n"
                "Content-Length: " + std::to_string(html.size()) + "\r\n"
                "Connection: close\r\n\r\n";

            send(client, header.data(), header.size(), 0);
            send(client, html.data(), html.size(), 0);
            close(client);
            break;  // done
        }
        close(sock);
    }).detach();

    return true;
}
```

A few design notes worth being explicit about:

**The race condition that isn't one.** `listen` is called before `open`, so the browser's TCP SYN is queued by the kernel even while you're still in `open`. By the time the thread calls `accept`, the connection is already waiting. No sleep/poll needed.

**Why capture `html` by value.** The detached thread outlives the call site. If you captured a `string_view` or reference you'd have a dangling pointer the moment the caller's HTML buffer goes out of scope. The move into the lambda is free if the caller passes an rvalue.

**Favicon handling.** Chrome/Safari will fire a second request for `/favicon.ico`. Without the loop you'd serve your HTML to that request and the actual page load would get a `Connection refused`. The loop exits after the first real request.

**The file-saving option.** Your existing `open_plot_in_browser(path)` is still the right function for that. You'd expose both, possibly with a flag or separate overload:

```cpp
// Saves to path and opens — user keeps the file.
bool open_plot_in_browser(const std::filesystem::path& path);

// Serves from memory — ephemeral, no file written.
bool open_plot_in_browser(std::string html);
```

One thing to flag: `std::system` is a bit ugly and has the usual shell-injection caveats — but since you're constructing the URL yourself from a `localhost:PORT` string it's not a real risk here, just aesthetically unpleasant. If that bothers you, `posix_spawn` or `execvp` is the alternative but it's more boilerplate for marginal gain in this context.

# claude response

Taking each question directly:

**Would it be quicker?** Probably not. The browser still has to make a TCP connection, do an HTTP round-trip, and parse the response. Opening a local file skips all of that. The server approach wins on cleanup cleanliness, not speed.

**Delete after a delay — how long?** The tricky part is that `std::system("open ...")` returns as soon as the OS has dispatched the event, not when the browser has read the file. On macOS with a running browser, the file read is nearly instant — 200ms is almost certainly enough. But cold-starting a browser could take 2–3 seconds. A conservative 3000ms covers essentially all real cases. The implementation is a detached thread with `std::this_thread::sleep_for`.

**On Unix you can also just unlink immediately.** If the browser already has an open file descriptor (which it will have within milliseconds of `open` returning), `unlink` removes the directory entry but the data persists until the last fd is closed. This is deterministic and clean — no guessing at timing. Doesn't work on Windows.

```cpp
bool open_plot_in_browser(const std::filesystem::path& path, bool keep = false) {
    std::string cmd = "open \"" + path.string() + "\"";
    if (std::system(cmd.c_str()) != 0) return false;

    if (!keep) {
        // On Unix: unlink immediately — browser already has or will get an fd.
        // The file data persists until the browser closes it.
        std::filesystem::remove(path);

        // On Windows you'd need the sleep approach instead:
        // std::thread([path]() {
        //     std::this_thread::sleep_for(std::chrono::milliseconds(3000));
        //     std::filesystem::remove(path);
        // }).detach();
    }
    return true;
}
```

**Which function instantiates a web server?** None — there's no web server library in that code. `socket()`, `bind()`, `listen()`, `accept()` are raw POSIX syscalls. The 30-odd lines in the lambda *are* the entire HTTP server. Memory overhead is one detached thread (default 8MB stack, immediately blocking on `accept`) plus the captured HTML string. For a typical Plotly payload of a few hundred KB to low MB, that's negligible.

Given all this, the delete-on-Unix approach is the simplest and most correct. The one-shot server is an interesting trick but adds real complexity for no user-visible benefit.

# claude response

Two options:

**`std::system` with `start`** — simple, no extra headers:
```cpp
std::system(("start \"\" \"" + path.string() + "\"").c_str());
```
The empty first arg is a required dummy window title — `start` treats the first quoted arg as the title, so without it a path with spaces gets misinterpreted.

**`ShellExecuteEx`** — proper Win32 API, more robust, doesn't spawn a cmd.exe intermediary:
```cpp
#include <windows.h>
#include <shellapi.h>

SHELLEXECUTEINFOA info{};
info.cbSize = sizeof(info);
info.fMask  = SEE_MASK_DEFAULT;
info.lpVerb = "open";
info.lpFile = path.string().c_str();
info.nShow  = SW_SHOWNORMAL;
ShellExecuteExA(&info);
```

`ShellExecute` is what Explorer uses internally when you double-click a file, so it's the most semantically correct equivalent to macOS `open`. The `start` approach works fine in practice but goes via cmd.exe which is unnecessary overhead.

For the deletion problem on Windows specifically — you're stuck with the sleep-and-delete thread since Windows file locking is mandatory, not advisory. The `FILE_FLAG_DELETE_ON_CLOSE` flag only helps if you hold the handle open yourself, which you don't after handing it to the browser. Three seconds is the pragmatic answer.

On the second: agreed, no good reason to offer `std::system` as an alternative if `ShellExecuteEx` is strictly better — no cmd.exe, no quoting edge cases, proper semantic meaning. I included it out of "here's the simple version first" habit, which wasn't useful here.

On the third: `SHELLEXECUTEINFOA` is a plain C struct, not a class — my mistake in calling it that. It's declared in `<shellapi.h>`, which I did list. `<windows.h>` must also be included as it's a prerequisite for essentially all Win32 headers. So the two includes I already showed are the complete answer:

```cpp
#include <windows.h>
#include <shellapi.h>
```

The `A` suffix is Windows naming convention for the ANSI (narrow string) variant of the struct, as opposed to `SHELLEXECUTEINFOW` which takes wide strings (`wchar_t*`). If your paths are UTF-8 `std::string` you'd typically convert to `std::wstring` and use the `W` variant, but for ASCII paths the `A` variant is fine.
